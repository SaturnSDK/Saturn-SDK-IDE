Artistic Style Supplemental Documentation

Problems with Pad Operators

This innocent looking and popular option is actually the cause of several formatting problems. The following are known problems with pad operators (--pad=oper / -p).

  • It removes extra spaces from formatted expressions.
    These formatted constants: 
    const COLORREF blue   = RGB(0,    0   , 0xff);
const COLORREF red    = RGB(0xff, 0,    0);
const COLORREF green  = RGB(0,    0xff, 0);
    Becomes this: 
    const COLORREF blue = RGB(0, 0, 0xff);
const COLORREF red = RGB(0xff, 0, 0);
const COLORREF green = RGB(0, 0xff, 0);
    And this formatted array: 
        {  blue,        "blue"   },
    {  red,         "red"    },
    {  green,       "green"  },
    Becomes this: 
        { blue, "blue" },
    { red, "red" },
    { green, "green" },
    And this formatted code: 
    case scrollbar:     sysColor = GetSysColor(COLOR_SCROLLBAR);      break;
case background:    sysColor = GetSysColor(COLOR_BACKGROUND);     break;
case activecaption: sysColor = GetSysColor(COLOR_ACTIVECAPTION);  break;
case menu:          sysColor = GetSysColor(COLOR_MENU);           break;
    Becomes this: 
    case scrollbar: sysColor = GetSysColor(COLOR_SCROLLBAR); break;
case background: sysColor = GetSysColor(COLOR_BACKGROUND); break;
case activecaption: sysColor = GetSysColor(COLOR_ACTIVECAPTION); break;
case menu: sysColor = GetSysColor(COLOR_MENU); break;
  • It will pad PARENS on the OUTSIDE. The option pad=paren pads only the inside???
    This: 
    if(i == 0)
    tabCount += 2;
    Becomes this (space after “if”): 
    if (i == 0)
    tabCount += 2;
  • It pads negative numbers that are not operators.
    This: 
    GoMessage(-1);
(-1)
    Becomes this: 
    GoMessage( -1);
( -1)
  • It sometimes adds an extra space after preprocessor directives.

Problems with Break Header Blocks, Break All Blocks

The following are known problems with the break header blocks and break all blocks options (--break-blocks, --break-blocks=all).
  • If a statement is NOT part of a block, the following statements are all double spaced. Statements enclosed in a block are formatted correctly.
    This: 
    visStatusHeight = rcStatus.bottom - rcStatus.top;
if(!opt.bViewStatus)
    visStatusHeight = 0;
GetClientRect(hwnd, &rcEdit);
iEditLeft = rcEdit.left;
iEditRight = rcEdit.right - iEditLeft;
    Becomes this (with extra lines added where they shouldn't be): 
    visStatusHeight = rcStatus.bottom - rcStatus.top;

if (!opt.bViewStatus) 
    visStatusHeight = 0;

GetClientRect(hwnd, &rcEdit);

iEditLeft = rcEdit.left;

iEditRight = rcEdit.right - iEditLeft;
  • Comments are broken from the block.
    This: 
    // line numbers and select must be done after SCI_STYLECLEARALL
if(setDialog)
    skip = true;
    Becomes this (with a line added after the comment): 
    // line numbers and select must be done after SCI_STYLECLEARALL

if(setDialog)
    skip = true;
